2. Service Details

Python/FastAPI Auth Service :8001

  • Database: chatbox_auth (users table)
  • Public: register, login, logout, ping
  • Internal: GET /auth/users/{id} and /by-username/{name}
  • Redis: Stores blacklisted JWT tokens on logout (TTL = token expiry)
  • Kafka: Produces auth.events (register/login/logout)

Go/Gin Chat Service :8003

  • Database: chatbox_chat (rooms, room_admins, muted_users)
  • REST: Room CRUD, admin/mute management, PM send
  • WebSocket: /ws/{roomId} (room chat), /ws/lobby (presence + DMs)
  • Kafka Producer: chat.messages, chat.private, chat.events
  • Kafka Consumer: file.events (broadcasts file notifications to WS)
  • HTTP Client: Calls Auth Service for user lookups

Python/FastAPI Message Service :8004

  • Database: chatbox_messages (messages table)
  • CQRS Write: Kafka consumer persists messages from chat.messages + chat.private
  • CQRS Read: REST API for history replay and room history
  • Idempotent: Skips duplicate message_id (UUID) on retry
  • DLQ: Failed messages go to chat.dlq after 3 retries
  • HTTP Client: Calls Auth Service to resolve usernames for DMs

Node/Express File Service :8005

  • Database: chatbox_files (Prisma ORM)
  • Upload: Multer, 150MB max, extension + MIME validation
  • Download: Stream from disk with RFC 5987 encoding
  • Security: Path traversal prevention, filename sanitization
  • Kafka: Produces file.uploaded to file.events (fire-and-forget)

4. Redis - Token Blacklist

sequenceDiagram
    participant U as User
    participant FE as Frontend
    participant Auth as Auth Service
    participant R as Redis

    Note over R: Only Auth Service uses Redis

    U->>FE: Click Logout
    FE->>Auth: POST /auth/logout (Bearer JWT)
    Auth->>R: SET blacklist:{token} = 1 (TTL = 3 hours)
    Auth-->>FE: 200 OK

    Note over R: On next request from any service...

    FE->>Auth: POST /auth/ping (Bearer JWT)
    Auth->>R: EXISTS blacklist:{token}?
    R-->>Auth: 1 (exists = blacklisted)
    Auth-->>FE: 401 Unauthorized

Why Redis for this?

JWTs are stateless - you can't "invalidate" them without a blacklist. Redis is perfect because: (1) O(1) lookups on every authenticated request, (2) TTL auto-cleans expired tokens - no cron jobs needed, (3) In-memory = microsecond latency. The blacklist only lives as long as the token would have been valid anyway.

5. Data Flows

Room Chat Message

sequenceDiagram
    participant FE as Frontend
    participant WS as Chat Service (WebSocket)
    participant K as Kafka
    participant MS as Message Service
    participant DB as chatbox_messages

    FE->>WS: WebSocket: {"type":"message","text":"Hello"}
    Note over WS: Rate limit check (30/10s)
    Note over WS: Mute check
    Note over WS: Generate UUID msg_id
    WS-->>FE: Broadcast to all room connections (REAL-TIME)
    WS->>K: Produce to chat.messages (ASYNC)
    K->>MS: Consumer picks up message
    Note over MS: Check if msg_id exists (idempotent)
    MS->>DB: INSERT message
    Note over FE,DB: On reconnect, Frontend GETs /messages/rooms/{id}?since=timestamp
Frontend WS --> Chat Service (validate + broadcast immediately) --> Kafka chat.messages --> Message Service --> PostgreSQL
Key insight: Real-time delivery is synchronous (instant). Persistence is asynchronous (eventual). Users see messages immediately even if Kafka is slow.

Why no foreign keys across databases?

Each service owns its data independently. Messages store sender_name and sender_id as denormalized copies - they don't reference the users table in chatbox_auth. This means services can be deployed, scaled, and even have their schemas changed independently without breaking each other. The trade-off is eventual consistency - if a username changes, old messages still show the old name.

Internal-only endpoints (NOT exposed through Kong)

GET /auth/users/{id} and GET /auth/users/by-username/{name} have no Kong route. They're only reachable over the Docker network by Chat Service and Message Service. This is your security boundary - external clients can never enumerate users.
